# 코드 정리 프롬프트

**목표**: 면접관/리뷰어가 빠르고 정확하게 이해할 수 있는 포트폴리오 문서 작성. (코드 스니펫 제외)

## 톤 앤 매너 라우팅 (TONE & MANNER ROUTING)

사용자가 입력한 태그에 따라 문서의 어조를 완벽하게 전환한다:

**[story] (스토리텔링 모드)**

- **목표**: 끝까지 읽고 싶은, 스토리텔링이 살아있는 포트폴리오.
- **문체**: 정중하고 부드러운 한국어 어조 ("~했습니다", "~합시다").
- **전개**: 목적과 로직을 한 편의 서사처럼 연결해 면접관이 기획 의도와 치열한 고민에 공감하게 만든다(Why/How 직결).

**[info] (직관적 정보 모드)**

- **목표**: 면접관이 1초 만에 핵심 성과를 스캐닝할 수 있는 레주메(이력서)/위키 스타일.
- **문체**: 객관적이고 짧은 개조식, 명확한 단문 ("~구현", "~함", "~달성").
- **전개**: 맥락이나 부연 설명을 배제하고 오직 성과 수치, 아키텍처 결론, 의사결정 팩트만 건조하고 날카롭게 나열한다.

---

## 구조

**[팀 프로젝트]**: 개요 → 스택 → 내 담당 파트 → 핵심 로직
**[개인 프로젝트]**: 목적 → 스택 → 핵심 기능 → 핵심 로직
**[공통 선택]**: 인사이트 (트러블슈팅 / 성능 최적화 / 회고)

- **개요 / 목적**: 목적, 기능, 팀/기간 기록.
- **스택**: 기술 목록 + 1줄 기술 선택 이유.
- **담당 파트 / 핵심 기능 [핵심]**: 구체적 기여 범위 / 해결한 문제.
- **핵심 로직 [핵심]**: 단계별 데이터 흐름, 의사결정 이유.
- **인사이트 [선택]**: 사용자가 분명하게 제공한 경우에만 작성한다. 절대 지어내지 말 것.
  - _트러블슈팅 (문제→원인→해결→결과)_
  - _성능 최적화 (개선 전/후 수치)_

## 단계 1 — Claude: 초안 작성

- **작업**: `src/code_workspace/{team|personal}/{project_name}/{project_name}.md` 초안 작성.
- **[핵심]** 섹션에 가장 큰 비중을 둔다.
- 레포지토리에 존재하는 검증된 사실만 작성한다. 과장 금지.
- 한국어로 작성 (RAG 등 영문 기술 용어는 원어 유지).
- **강제**: 사용자가 요청한 태그(`[story]` 혹은 `[info]`)의 톤 앤 매너를 문서 전체에 똑같이 적용한다.
- Notion 토글용 계층 구조를 명확히 잡는다.

## 단계 2 — Gemini: 보완

- **작업**: `@{project_name}.md`를 읽고 면접관 관점에서 문서 품질 보완.
- **[핵심]** 섹션 강화: 모호한 표현을 수치나 정확한 기술명으로 교체한다.
- 중요한 성과/수치는 `> `나 `!> ` 로 강조해 시각적 지루함을 깬다.
- **강제**: 사용자가 요청한 태그(`[story]` 혹은 `[info]`)의 문체가 처음부터 끝까지 흐트러짐 없이 유지되는지 검증하고 철저히 교정한다.
- 사용자가 주지 않은 인사이트를 절대 혼자 지어내지 않는다 (No hallucination).

## 단계 3 — Claude: 최종 검증 및 보완

- **작업**: `@{project_name}.md`의 어조 및 팩트 최종 교정.
- 인사이트가 있다면 수치나 구조가 올바른지 검증한다.
- **강제**: 요청된 톤(`[story]` 혹은 `[info]`)과 일치하지 않는 어색한 문맥을 모조리 잡아 수정한다.
- 과장되거나 입증 불가한 표현, 오탈자를 수정한다.
