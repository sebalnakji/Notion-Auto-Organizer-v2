# 개념 정리 프롬프트

## 목표

아무것도 모르는 학생에게 처음 가르치는 선생님처럼 개념을 쉽고 빠르고 정확하게 설명한다.

---

## 구조

필수: Why → Where → What → How
선택: Pitfall, So What (오해가 잦거나 심화 안내가 필요할 때만)

- Why: 이 개념이 등장한 배경과 해결한 문제
- Where: 실제 사용되는 상황 2~3가지
- What: 정의 + 실생활 비유 (핵심, 비유가 불충분하면 추가)
- How: 단계별 작동 방식 + 실제 사용 예시 (핵심)
- Pitfall: 입문자가 흔히 빠지는 함정 2~3가지
- So What: 연결되는 다음 학습 주제

What과 How가 가장 중요하다. 이 두 섹션이 불명확하면 전체가 흔들린다.

---

## 단계 1 — Claude: 초안 작성

```
주제: {주제명}
작업: 개념 정리 초안 작성 후 src/concept_workspace/{대주제}/{중주제}/{개념명}.md 저장

구조를 순서대로 따른다.
- What, How에 가장 많은 비중을 둔다.
- How는 반드시 실제 예시(코드/시나리오/다이어그램 중 적합한 것)를 포함한다.
- Pitfall, So What은 필요한 경우에만 포함한다.

조건:
- 검증된 사실만 포함한다. 불확실한 내용은 제외한다.
- Notion 토글 변환이 쉽도록 계층 구조를 명확히 한다.
- 공식 문서나 신뢰할 수 있는 링크가 있으면 포함한다.
- 작성 후 스토리 흐름과 팩트를 자체 검토하고 저장한다.
- 문서는 한글로 작성한다. 단, RAG, LLM 등 전문 용어와 고유명사는 원문 그대로 사용한다.
```

## 단계 2 — Gemini: 보완

```
파일: @{개념명}.md
작업: 스토리 흐름을 유지하며 문서 품질을 높인다.

점검 순서:
1. Why: 역사적·기술적 배경이 충분한가? 부족하면 추가한다.
2. Where: 사례가 구체적이고 다양한가? 보강한다.
3. What [최우선]: 비유가 직관적인가? 전문 용어가 설명 없이 쓰였는가? 부족하면 교체하거나 추가한다.
4. How [최우선]: 단계별 설명이 명확한가? 실제 예시가 있는가? 없으면 추가한다.
5. Pitfall (있을 때만): 검증되지 않은 내용은 삭제한다.
6. So What (있을 때만): 레퍼런스 링크를 추가한다.

조건:
- 불확실한 내용은 삭제한다.
- Why → Where → What → How 흐름이 끊기지 않도록 재배치한다.
- 중요 정보는 `> ` 또는 `!> `로 강조한다.
- 불필요한 수식어를 제거하고 간결하게 유지한다.
- 검토 후 저장한다.
```

## 단계 3 — Claude: 최종 검증 및 보완 + Notion 업로드

```
파일: @{개념명}.md
작업: 최종 검증 후 Notion 업로드 (업로드 규칙은 main_prompt.md 참조)
내용 확인 후 보완할 점이 있으면 보완한다.

검증:
- Why/Where: 흐름이 자연스럽게 연결되는가?
- What [최우선]: 비유가 명확하고 초보자도 즉시 이해할 수 있는가?
- How [최우선]: 단계별 설명이 구체적이고 실제 예시가 있는가?
- Pitfall/So What (있을 때만): 내용이 정확하고 유효한가?
- 오해를 부르는 문장, 입증 불가한 주장, 유효하지 않은 링크를 제거한다.
- 오탈자와 어색한 문맥을 수정한다.
```
