# 코드 정리 프롬프트 (포트폴리오용)

## 목표

면접관/리뷰어가 프로젝트를 빠르고 정확하게 이해할 수 있도록 핵심 로직 위주로 정리한다.
코드 스니펫 없이 로직 흐름과 의사결정 이유 중심으로 작성한다.

---

## 구조

### 팀 프로젝트

필수: 프로젝트 개요 → 개발 스택 → 내 담당 파트 → 핵심 로직
선택: 인사이트 (사용자가 직접 내용을 제공한 경우에만)

- 프로젝트 개요: 목적과 주요 기능 1~2줄 요약, 팀 규모·개발 기간
- 개발 스택: 기술 목록 + 핵심 기술 선택 이유 한 줄
- 내 담당 파트: 담당 기능·모듈, 기여 범위 (핵심)
- 핵심 로직: 구현 흐름 단계별 설명, 의사결정 이유, 데이터 흐름 (핵심)
- 인사이트: 트러블슈팅 / 성능 최적화 / 프로젝트 소감

### 개인 프로젝트

필수: 만든 목적 → 개발 스택 → 핵심 기능 → 핵심 로직
선택: 인사이트 (사용자가 직접 내용을 제공한 경우에만)

- 만든 목적: 어떤 문제를 해결하기 위해 만들었는지
- 개발 스택: 기술 목록 + 핵심 기술 선택 이유 한 줄
- 핵심 기능: 주요 기능 목록 + 각 기능이 해결하는 문제 (핵심)
- 핵심 로직: 동작 흐름 단계별 설명, 의사결정 이유, 데이터 흐름 (핵심)
- 인사이트: 트러블슈팅 / 성능 최적화 / 프로젝트 소감

담당 파트/핵심 기능과 핵심 로직이 가장 중요하다. 면접관이 가장 주목하는 섹션이다.

인사이트 규칙:

- 코드만으로는 알 수 없다. 사용자가 직접 내용을 전달한 경우에만 작성한다.
- 전달받지 않았으면 반드시 생략한다.
- 트러블슈팅: 문제 상황 → 원인 → 해결 → 결과
- 성능 최적화: 개선 전/후 수치 필수
- 프로젝트 소감: 배운 점, 아쉬운 점, 개선 방향

## 단계 1 — Claude: 초안 작성

```
레포: {레포명}
유형: {팀 프로젝트 / 개인 프로젝트}
작업: 포트폴리오용 초안 작성 후 src/code_workspace/{team or personal}/{프로젝트명}/{프로젝트명}.md 저장

구조를 순서대로 따른다.
- 담당 파트/핵심 기능, 핵심 로직에 가장 많은 비중을 둔다.
- 핵심 로직은 입력 → 처리 → 출력 흐름으로 설명한다.
- 인사이트는 사용자가 내용을 제공한 경우에만 작성한다.

조건:
- 레포의 실제 내용만 기반으로 작성한다. 추측과 과장은 제외한다.
- Notion 토글 변환이 쉽도록 계층 구조를 명확히 한다.
- 작성 후 흐름과 팩트를 자체 검토하고 저장한다.
- 문서는 한글로 작성한다. 단, RAG, LLM 등 전문 용어와 고유명사는 원문 그대로 사용한다.
```

## 단계 2 — Gemini: 보완

```
파일: @{프로젝트명}.md
작업: 면접관 관점에서 문서 품질을 높인다.

팀 프로젝트 점검:
1. 프로젝트 개요: 목적이 한눈에 파악되는가?
2. 개발 스택: 기술 선택 이유가 납득 가능한가?
3. 내 담당 파트 [최우선]: 기여 범위가 구체적인가? 모호한 표현은 수치나 기능명으로 교체한다.
4. 핵심 로직 [최우선]: 흐름이 명확한가? 의사결정 이유가 설득력 있는가?
5. 인사이트 (있을 때만): 사용자가 제공한 내용만 기반으로 보완한다. 임의로 추가하지 않는다.

개인 프로젝트 점검:
1. 만든 목적: 동기가 구체적이고 공감 가능한가?
2. 개발 스택: 기술 선택 이유가 납득 가능한가?
3. 핵심 기능 [최우선]: 각 기능이 해결하는 문제가 명확한가? 나열식이면 문제 해결 관점으로 재구성한다.
4. 핵심 로직 [최우선]: 흐름이 명확한가? 의사결정 이유가 설득력 있는가?
5. 인사이트 (있을 때만): 사용자가 제공한 내용만 기반으로 보완한다. 임의로 추가하지 않는다.

조건:
- 레포 내용과 다르거나 과장된 표현은 수정한다.
- 불필요한 설명은 제거한다. 포트폴리오는 간결할수록 좋다.
- 핵심 수치나 성과는 `> ` 또는 `!> `로 강조한다.
- 검토 후 저장한다.
```

## 단계 3 — Claude: 최종 검증 + Notion 업로드

```
파일: @{프로젝트명}.md
작업: 최종 검증 후 Notion 업로드 (업로드 규칙은 main_prompt.md 참조)

검증:
- 개요/목적·스택: 한눈에 파악되는가?
- 담당 파트/핵심 기능 [최우선]: 기여가 구체적이고 명확한가?
- 핵심 로직 [최우선]: 흐름이 명확하고 의사결정 이유가 설득력 있는가?
- 인사이트 (있을 때만): 사용자 제공 내용만 포함됐는가? 트러블슈팅 구조, 최적화 수치, 소감 구체성을 확인한다.
- 과장·오류·입증 불가 내용을 제거한다.
- 유효하지 않은 링크를 제거하거나 교체한다.
- 오탈자와 어색한 문맥을 수정한다.
```
